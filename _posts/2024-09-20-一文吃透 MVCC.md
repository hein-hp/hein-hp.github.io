## 前言

这篇文章我们准备聊一聊 MySQL 的 MVCC 机制，MVCC 是 MySQL 中的一种高效、不加锁并发的机制，通过记录的版本管理实现事务的隔离性，允许读写操作同时进行，提高数据库的并发性能和响应能力。

在了解 MVCC 之前，我们还需要知道 MySQL 中的事务、并发事务可能出现的问题，以及在 SQL 标准中为了解决并发事务的问题规定了哪些事务隔离级别。

<!--more-->

## 什么是事务
事务是指作为单个逻辑工作单元执行的一系列操作。这些操作通常包括数据的读取、写入或修改。

这里比较重要的是事务的 ACID 四大特性。

+ 原子性 Atomicity：一个事务是一个不可分割的工作单元。事务中的所有操作要么全部完成，要么全部不发生，不会停留在中间某个状态。如果在执行事务过程中出现异常，整个事务都将回滚到事务执行前的状态。
+ 一致性 Consistency：事务应确保数据库从一个一致性状态转换到另一个一致性状态。简单来说，事务执行完成后，数据库中的所有数据都必须符合数据完整性约束。
+ 隔离性 Isolation：并发事务的执行过程中，一个事务不应该受到其他事务的影响。这意味着即使多个事务同时进行，每个事务看起来都是在独立地执行。
+ 持久性 Durability：一旦事务被提交，它对数据库的变更将是持久的。即使系统发生故障，数据也不会丢失。

在 InnoDB 中，undo log 可以保证事务的原子性，redo log 可以保证事务的持久性，而隔离性则是由 MVCC 以及锁机制来保证的，最后一致性则是由其余三者共同保证的。

我们这篇文章其实重点讨论的就是关于隔离性的问题。

## 并发事务问题
我们知道，MySQL 客户端支持多个客户端连接，这意味着 MySQL 会出现同时处理多个事务的情况。这时就可能出现并发事务相关的问题，这包括脏读、不可重复读、幻读。

+ 脏读 (Dirty Read)：一个事务读取了另一个尚未提交的事务写入的数据。如果那个事务随后被回滚，那么第一个事务就基于了一个实际上从未真正存在的值进行了操作。
+ 不可重复读 (Non-repeatable Read)：在一个事务内，多次读取同一数据得到的结果不同。这种情况发生在当一个事务重新读取之前读过的数据时，发现其他事务已经修改了该数据并提交了更改。
+ 幻读 (Phantom Read)：类似于不可重复读，但这里的问题是在查询结果集上。如果一个事务执行两次相同的查询，第二次查询返回了一组与第一次不同的行（即新增或删除了某些行），这可能是因为另一个事务在这两次查询之间插入或删除了满足查询条件的记录。

这里不可重复读强调 **某条记录的数据发生了更改**，而幻读强调 **记录的数量发生了变化**。

## 事务隔离级别
为了解决并发事务的问题，SQL 标准又提出了四种事务隔离级别，事务隔离级别定义了事务之间的可见性规则，也就是一个事务对数据的更改在什么条件下对其他并发事务可见。

这四种隔离级别提供了不同程度的数据一致性和并发控制。

| 隔离级别 | 说明 | 可能出现的问题 |
| --- | --- | --- |
| 读未提交 | 一个事务可以读取到其他事务尚未提交的数据 | 脏读、不可重复读和幻读 |
| 读提交 | 一个事务只能读取到其他事务已经提交的数据 | 不可重复读和幻读 |
| 可重复读 | 一个事务多次读取同一数据时，得到的结果是一致的 | 幻读 |
| 串行化 | 强制所有事务串行排队执行 |  |


所以，要解决脏读现象，就要升级到读提交以上的隔离级别；要解决不可重复读现象，就要升级到可重复读的隔离级别，但是要解决幻读现象不建议将隔离级别升级到串行化，这会严重影响 MySQL 的性能。

在 SQL 标准中，虽然可重复读隔离级别会出现幻读问题，但在 MySQL 中可重复读隔离级别可以在很大程度上避免幻读问题，当然并不是完全解决。

+ 针对快照读，主要是通过 MVCC 解决的，我们后面详细再说。
+ 针对当前读，主要是通过加锁（临键锁），锁住插入间隙解决的，具体可以参考 [一文吃透 MySQL 中的锁](https://hein-hp.github.io/mysql/2024/09/18/%E4%B8%80%E6%96%87%E5%90%83%E9%80%8F-MySQL-%E4%B8%AD%E7%9A%84%E9%94%81/#)。

那么什么是快照读和当前读呢？

## 快照读和当前读
在 MySQL 中，特别是在 InnoDB 存储引擎中，有两种主要的读取方式：当前读和快照读。这两种读取方式与事务隔离级别密切相关，影响着数据的一致性和并发性能。

### 快照读
快照读是基于事务开始时的数据状态来读取数据，这意味着它不会看到其他事务在此之后所做的更改，即使这些更改已经提交。

快照读主要是依赖于 MVCC 实现的，它允许每个事务在事务期间看到一致的视图，这个视图反映了事务开启时的数据状态，快照读的实现，或者说 MVCC 的实现可以避免并发事务的读写导致的锁定。

快照读主要发生在 RC 和 RU 隔离级别下的普通 select 语句。

### 当前读
当前读总是基于最新的已提交的数据，这意味着它可以看到其他事务最新提交的所有更改。

当前读主要是基于 MySQL 锁机制实现的，这样 MySQL 会在执行读写操作时获取共享锁或者排他锁，来确保数据的一致性。

当前读主要发生在执行锁定的 SQL 中，并不仅限于具体的事务隔离级别，这些 SQL 主要包括：

+ select ... for update、select ... lock in share mode、update、delete、insert

而在 SERIALIZABLE 隔离级别下，由于各个事务严格串行执行，所以即使是普通的 select 语句也属于当前读，而在 RU 隔离级别下，普通的 select 语句由于可以直接读到最新的数据，即使其他事务还未提交，所以我认为也属于当前读。

那么这样一看，只有在 RC 和 RR 隔离级别下的普通 select 才属于快照读，而其他的数据读写都属于当前读。

了解了一些前置相关的内容，接下来就可以正式进入 MVCC。

## MVCC
首先我们要知道，MVCC 的实现主要依赖三个东西，行记录的隐藏字段、undo log 以及 Read View。

### 隐藏字段
对于使用 InnoDB 存储引擎的数据库表，它的聚簇索引记录中都包含额外的三个隐藏列：

+ DB_TRX_ID：记录了插入或更新该行的最后一个事务的事务 id。
+ DB_ROLL_PTR：指向回滚段中的 undo log 记录，用于构造旧版本的数据。
+ DB_ROW_ID：如果表没有主键或唯一索引，InnoDB 会自动生成一个 6 字节的隐藏列 DB_ROW_ID 作为聚簇索引。

### Undo log 版本链
undo log 日志是 MVCC 实现的关键所在，它也是 MySQL 事务回滚时的依据。

不同事务或者相同事务对同一记录行的修改，会使该记录行的 undo log 串成链表，链表头就是最新的记录，链表尾就是最早的记录，这样就形成了 undo log 版本链。

更详细的可以参考 MySQL 日志部分的内容。

### Read View
Read View 主要是用来判断 undo log 版本链中的哪个版本对当前事务是可见的。

其实本质就是事务在进行快照读操作时生成的读视图（Read View），在该事务执行快照读的那一刻，会生成数据库系统当前的一个快照，记录并维护系统当前活跃事务的 id（每个事务开启时，都会被分配一个自增 id）。

当然，Read View 只针对 RC 和 RR 级别，而对于 RU 和 Serializable 隔离级别，

+ 在 RU 隔离级别下，事务可以读取其他事务尚未提交的数据，即脏读。这意味着不需要通过 Read View 来限制访问范围，事务可以自由地读取其他事务的未提交数据。由于没有对可见性进行严格控制，因此不需要创建或使用 Read View。
+ 在 Serializable 隔离级别下，事务具有最高的隔离性，确保每次读取都能看到一致的快照。为了实现这种隔离级别，MySQL 使用锁机制来保证事务之间的串行执行。由于事务按顺序执行，并且不允许并发操作，所以也不需要使用 Read View 进行可见性判断。

这里注意一点，执行开启事务的命令，并不意味着启动了事务。在 MySQL 有两种开启事务的命令，分别是：

+ 第一种：begin/start transaction；
+ 第二种：start transaction with consistent snapshot；

这两种开启事务的命令，事务的启动时机是不同的：

+ 执行了 begin/start transaction 后，并不代表事务启动，只有在执行这个命令后，执行了第一条 SQL 语句，才是事务真正启动的时机；
+ 执行了 start transaction with consistent snapshot，就会马上启动事务。

#### Read View 可见性原则
Read View 会遵循一个可见性原则，将要被修改的数据的 DB_TRX_ID 与系统当前其他活跃事务的 id 去对比。

如果 DB_TRX_ID 跟 Read View 的属性做了某些比较，不符合可见性，就会通过 DB_ROLL_PTR 回滚指针去取出 undo log 中的 DB_TRX_ID 再比较。

也就是遍历链表的 DB_TRX_ID （从链首到链尾，即从最近的一次修改查起），直到找到满足特定条件的 DB_TRX_ID，那么该 DB_TRX_ID 所在的记录就是当前事务能看见的最近的版本。

具体来说，Read View 会维护下面四个重要属性：

|  字段 | 含义  |
| --- | --- |
| creator_trx_id | 创建该 Read View 的事务的事务 id |
| trx_ids | 创建 Read View 时，数据库中 **活跃事务** 的事务 id 集合，“活跃事务”指的是，开启但还未提交的事务 |
| up_limit_id | 创建 Read View 时，当前数据库中活跃事务中 **事务 id 最小的事务**，即 trx_ids 的最小值 |
| low_limit_id | 并不是 trx_ids 的最大值，而是创建 Read View 时当前数据库中应该分配的下一个事务的 id 值，也就是 **全局事务中最大的事务 id 值 + 1** |


Read View 的可见性判断如下：

+ 如果记录的 DB_TRX_ID 与 creator_trx_id 一致，表示当前事务正在访问自己修改的记录，该版本记录可见。
+ 如果记录的 DB_TRX_ID 值小于 Read View 中的 up_limit_id 值，表示该版本的记录是在创建 Read View **前** 已经提交的事务生成的，所以该版本的记录对当前事务 **可见**。
+ 如果记录的 DB_TRX_ID 值大于等于 Read View 中的 low_limit_id 值，表示这个版本的记录是在创建 Read View **后** 才启动的事务生成的，所以该版本的记录对当前事务 **不可见**。
+ 如果记录的 DB_TRX_ID 值在 Read View 的 up_limit_id 和 low_limit_id 之间，则进一步判断 DB_TRX_ID 是否在 trx_ids 列表中： 
    - 如果记录的 DB_TRX_ID **在** trx_ids 列表中，表示生成该版本记录的事务依然活跃（还没提交事务），所以该版本的记录对当前事务 **不可见**。
    - 如果记录的 DB_TRX_ID **不在** trx_ids 列表中，表示生成该版本记录的活跃事务已经被提交，所以该版本的记录对当前事务 **可见**。

这里比较疑惑的点是为啥 up_limit_id 是最低水位，low_limit_id 是最高水位？老外的方向是反的？

### RR 和 RC 下的 Read View
RC 和 RR 下生成 Read View 的时机是有所差异的：

+ RC：每次 SELECT 快照读前都生成一个全新的 ReadView。
+ RR：启动事务后第一次 SELECT 快照读之前生成一个 Read View，后续整个事务期间都沿用同一个 Read View。

正因为 RC 和 RR 生成 Read View 的时机不同，才会导致两个级别下看到的数据会不一致。

下面我们举个例子。

假设一行数据的初始状态如下：

![image-20240920083446572](/img/image-20240920083446572.png)

事务 A 和事务 B 按照如下的顺序执行：

|  事务 A |  事务 B |
| --- | --- |
| begin; |  |
|  | begin; |
|  | select name from t where id = 1; |
| update t set name = 'pika' where id = 1; |  |
|  | select name from t where id = 1; |
| commit; |  |
|  | select name from t where id = 1;<br/>commit; |


#### RR
我们首先看在 RR 隔离级别下的执行情况。

首先事务 A 和事务 B 在开启之后生成的 Read View 如下：

|   |  事务 A |  事务 B |
| --- | --- | --- |
| creator_trx_id | 假设是 51 | 52 |
| trx_ids | [51] | [51, 52] |
| up_limit_id | 51 | 51 |
| low_limit_id | 52 | 53 |


+ 在事务 A 的 Read View 中，它的事务 id 是 51，由于它是第一个启动的事务，所以此时活跃事务的事务 id 列表就只有 51，活跃事务的事务 id 列表中最小的事务 id 是事务 A 本身，下一个事务 id 则是 52。
+ 在事务 B 的 Read View 中，它的事务 id 是 52，由于事务 A 是活跃的，所以此时活跃事务的事务 id 列表是 51 和 52，活跃的事务 id 中最小的事务 id 是事务 A，下一个事务 id 应该是 53。

接着我们看按照顺序执行 SQL 会发生什么？

事务 B 第一次读取 id 为 1 的 name，在找到记录后，会先比对这条记录的 DB_TRX_ID，发现 DB_TRX_ID 为 1，比事务 B 的 Read View 中的 up_limit_id 值（51）还小，这意味着修改这条记录的事务早就在事务 B 启动前就提交了，所以该版本的记录对事务 B 可见的，事务 B 可以获取到这条记录。

接着，事务 A 修改了这条记录（还未提交事务），此时数据库会对该行加排他锁，然后将该行数据写到 undo log 作为旧版本记录，形成 **版本链**，如下图：

![image-20240920083501498](/img/image-20240920083501498.png)

当事务 B 第二次读取时，再比对记录的 DB_TRX_ID，为 51，在事务 B 的 Read View 中 up_limit_id 和 low_limit_id 之间，进一步判断在 trx_ids 范围内，所以该版本记录对事务 B 是不可见的，所以事务 B 会沿着 undo log 版本链找到 DB_TRX_ID 为 1 的记录，所以读到的 name 值依然为 hein。

最后，当事务 A 提交之后，由于事务 B 的 Read View 在整个事务期间只会生成一次，所以分析过程还是像上面那么，最终会沿着 undo log 版本链找到 DB_TRX_ID 为 1 的记录，读到的 name 值依然为 hein。

#### RC
在 RC 隔离级别下的区别在于事务 B 每一次读取都会生成新的 Read View。

首先，事务 B 第一次读取时创建的 Read View 如下：

|   |  事务 A |  事务 B |
| --- | --- | --- |
| creator_trx_id | 假设是 51 | 52 |
| trx_ids | [51] | [51, 52] |
| up_limit_id | 51 | 51 |
| low_limit_id | 52 | 53 |


接着事务 A 更新 name 但还未提交，形成 undo log 版本链，

![image-20240920083506745](/img/image-20240920083506745.png)

接着事务 B 第二次读取，创建新的 Read View

|   |  事务 A |  事务 B |
| --- | --- | --- |
| creator_trx_id | 假设是 51 | 52 |
| trx_ids | [51] | [51, 52] |
| up_limit_id | 51 | 51 |
| low_limit_id | 52 | 53 |


由于事务 A 并未提交，所以这次事务 B 创建的新的 Read View 和之前的一样，接着事务 A 提交，事务 B 再次读取，又会创建新的 Read View，如下：

|   |  事务 A |  事务 B |
| --- | --- | --- |
| creator_trx_id | 假设是 51 | 52 |
| trx_ids | [51] | [52] |
| up_limit_id | 51 | 52 |
| low_limit_id | 52 | 53 |


可以看到，trx_ids 和 up_limit_id 都因为事务 A 的提交而发生了变化。

而第三次事务 B 再次读取记录时，在找到 id = 1 的这条记录时，发现这条记录的 DB_TRX_ID 是 51，比事务 B 的 Read View 中的 up_limit_id 值（52）还小，这意味着修改这条记录的事务早就在创建 Read View 前提交过了，所以该版本的记录对事务 B 是可见的，查询出的 name 值为 pika。

## RR 下的幻读问题
最后我们再来看一下在 RR 隔离级别下可能出现的幻读问题。

通过前面的例子，我们也看到了，在 RR 隔离级别下，由于 MVCC 的存在，如果其他事务插入了新的行，在同一个事务中多次查询，这些新增的行将不会被包含在当前事务的一致性视图中，这为快照读避免了幻读问题。

同时在当前读的场景中，由于会加临键锁，锁住插入的间隙，所以也可以避免幻读问题。

但是我们刚才讨论的要么全是快照读，要么全是当前读，那有没有考虑过快照读和当前读在一个事务内同时发生呢？

比如下面的例子：

|  事务 A |  事务 B |
| --- | --- |
| begin; | begin;<br/>select * from t_user; |
| insert into t_user(id, name) values(2, 'hein');<br/>commit; |  |
|  | select * from t_user for update; |
|  | commit; |


由于当前读总是基于最新的已提交的数据，所以在事务 B 中在第二次查询中也查询出了 id = 2 的记录。

这就发生了幻读的现象。

那么我们如何避免幻读问题呢？

首先，想要彻底解决幻读问题，一定要在 SERIALIZABLE 隔离级别才行，其次，如果想要在最大程度上避免幻读问题，可以退而求其次，使用 RR 隔离级别。

在 RR 隔离级别下，能使用快照读就尽量使用快照读，这样不仅可以减少锁冲突，提升并发度，而且还能避免幻读的发生。

如果在并发场景中一定要加锁？那就一定要在事务一开始就立即加锁，这样就会在相应位置插入间隙锁，也能有效的避免幻读的发生。
