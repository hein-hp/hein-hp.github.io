## 前言

MySQL 中的锁的种类有很多，在 InnoDB 存储引擎中支持了行锁，而行锁的加锁规则还是挺复杂的，这篇文章我结合《小林 Coding》和《MySQL 45 讲》的一些例子，以及我自己的一些看法和理解整理了 MySQL 中常见的锁机制，希望对你有所帮助。

<!--more-->

在 MySQL 中，按照锁的粒度，主要分为以下三类：

+ 全局锁：锁定数据库中的所有表。
+ 表级锁：每次操作锁住整张表。
+ 行级锁：每次操作锁住对应的行数据。

## 全局锁
使用下面的 SQL 可以锁住整个数据库。

```sql
flush tables with read lock
```

执行后，**整个数据库就处于只读状态了**，这时其他线程执行以下操作，都会被阻塞：

+ 对数据的增删改操作，比如 insert、delete、update 等语句。
+ 对表结构的更改操作，比如 alter table、drop table 等语句。

如果要释放全局锁，则要执行这条命令：

```sql
unlock tables
```

当然，一旦会话断开，全局锁就会被自动释放。

### 用途
全局锁主要应用于做 **全库逻辑备份**，这样在备份数据库期间，不会因为数据或表结构的更新，而出现备份文件的数据与预期的不一样。

### 特点
全局锁是一个重量级的操作，加上全局锁，意味着整个数据库都是只读状态。

存在以下问题：

+ 如果在主库上备份，在备份期间都不能执行更新，业务基本上就得中断。
+ 如果在从库上备份，在备份期间从库不能执行主库同步过来的二进制日志（binlog），会导致主从延迟。

**那么如何避免呢**

如果存储引擎支持 **可重复读的隔离级别**，那么我们可以使用 MySQL 的备份工具 mysqldump 并且加上`–single-transaction`参数来实现一个一致性的非阻塞的快照备份。

比如我们要备份 test 库的 t 表。

```java
mysqldump -u username -p test t --single-transaction > t_backup.sql
```

在执行备份时，首先 mysqldump 会开启一个新的事务，进而创建一个 Read View，后续事务执行期间都使用该 Read View，这样备份期间备份的数据一直是在开启事务时的数据，而且由于 MVCC 的支持，备份期间业务依然可以对数据进行更新，当 mysqldump 完成了所有需要备份的数据的读取之后，它会提交事务。此时，备份过程就完成了。

InnoDB 存储引擎默认的事务隔离级别正是可重复读，因此可以采用这种方式来备份数据库，但对于 MyISAM 这种不支持事务的引擎，在备份数据库时就要使用全局锁的方法。

## 表级锁
### 表锁
先来说说 **表锁**。

如果我们想对学生表（t_student）加表锁，可以使用下面的命令：

```sql
# 表级别的共享锁，也就是读锁
lock tables t_student read;

# 表级别的独占锁，也就是写锁
lock tables t_stuent write;
```

表锁除了会限制其他线程的读写外，也会限制本线程接下来的操作对象。

比如，A 线程执行

```sql
lock tables t1 read, t2 write;
```

则其他线程写 t1、读写 t2 都会被阻塞，同时在线程 A 释放锁之前，也只能执行读 t1、读写 t2 的操作，不能写 t1，自然也不能访问其他的表。

要释放表锁，可以使用下面这条命令，会释放当前会话的所有表锁：

```sql
unlock tables
```

另外，一旦会话退出后，也会自动释放所有表锁。

不过尽量避免在使用 InnoDB 存储引擎的表使用表锁，因为表锁的颗粒度太大，会影响并发性能，而 **InnoDB 实现了锁粒度更细的行级锁**。

### 元数据锁
元数据锁 MDL（Metadata Lock）

我们不需要显示的使用 MDL，因为当对数据库表进行操作时，会自动给这个表加上 MDL：

+ 对表记录进行 CRUD 操作时，会加上 **MDL 读锁**
+ 对表做结构变更操作时，会加上 **MDL 写锁**

MDL 是为了保证当线程对表执行 CRUD 操作时，防止其他线程对这个表结构做了变更。

读锁之间不互斥，因此可以有多个线程同时对一张表进行 CRUD。

当有线程在执行 CRUD 语句（加 MDL 读锁）的期间，如果有其他线程要更改该表的结构（申请 MDL 写锁），将会被阻塞，直到执行完 CRUD 语句（释放 MDL 读锁）。

反之，当有线程对表结构进行变更（加 MDL 写锁）的期间，如果有其他线程执行了 CRUD 操作（申请 MDL 读锁），也会被阻塞，直到表结构变更完成（释放 MDL 写锁）。

#### 长事务问题
**MDL 的释放时机**

MDL 在事务提交后才会释放，这意味着 **在事务执行期间，MDL 是一直持有的**，这在长事务情况下可能会发生意想不到的结果。

比如下面这个顺序的场景：

1. 首先，线程 A 开启事务（一直不提交），然后执行一条 select 语句，此时对该表加 MDL 读锁
2. 然后，线程 B 也执行了 select 语句，此时并不会阻塞，因为 MDL 读读并不冲突
3. 接着，线程 C 修改了表字段，由于线程 A 的事务并没有提交，持续占用 MDL 读锁，这时线程 C 无法申请到 MDL 写锁，就会被阻塞
4. 在线程 C 阻塞后，后续对该表的 select 语句都会被阻塞，如果此时有大量该表的 select 语句的请求到来，就会有大量的线程被阻塞住，这时数据库的线程很快就会爆满了

**为什么线程 C 因为申请不到 MDL 写锁，而导致后续的申请 MDL 读锁的查询操作也会被阻塞？**

申请 MDL 锁的线程会形成一个队列，MySQL 会按照队列中的顺序处理 MDL 请求，当一个写请求正在等待时，新的读请求会加入队列（阻塞），而不是直接执行，主要是为了避免写请求的饥饿问题。

所以为了能安全的对表结构进行变更，在对表结构变更前，先要看看数据库中的长事务，是否有事务已经对表加上了 MDL 读锁，如果有，则考虑 kill 掉这个长事务，然后再做表结构的变更。

### 意向锁
接着，说说 **意向锁**。

在 InnoDB 引擎中， 

+ 对记录加上共享锁（行锁）之前，需要先在表级别加上一个意向共享锁
+ 对记录加上独占锁（行锁）之前，需要先在表级别加上一个意向独占锁

也就是，当执行增、删、改操作，需要先对表加上意向独占锁，然后对该记录加独占锁，而普通的 select 是不会加行级锁的，因为普通的 select 语句是利用 MVCC 实现一致性读，是无锁的。

不过，select 也是可以对记录加共享锁和独占锁的，具体方式如下：

```sql
# 先在表上加上意向共享锁，然后对读取的记录加共享锁
select ... lock in share mode;

# 先表上加上意向独占锁，然后对读取的记录加独占锁
select ... for update;
```

意向共享锁和意向独占锁是表级锁，不会和行级的共享锁和独占锁发生冲突，并且意向锁之间也不会发生冲突，只会和共享表锁（lock tables ... read）和独占表锁（lock tables ... write）发生冲突。

由于表锁和行锁是读读共享、读写互斥、写写互斥的，如果没有意向锁，那么在加独占表锁时，需要遍历表里所有记录，查看是否有记录加了行锁，这样效率会很慢。

而有了意向锁，在对记录加独占锁前，先会加上表级别的意向独占锁，那么在加独占表锁时，直接查该表是否有意向独占锁，如果有就意味着表里已经有记录被加了独占锁，这样就不用去遍历表里的记录。

所以，**意向锁的目的是为了快速判断表中是否有记录被加锁。**一旦事务提交了，意向共享锁、意向排他锁，都会自动释放。

### 自增锁
自增锁是 MySQL 针对 AUTO_INCREMENT 自增主键而设计的一种可配置的锁机制，可以显著提高 insert 语句的性能。

在 MySQL 8.0 之后，配置的变量是 innodb_autoinc_lock_mode。

在介绍自增锁之前，我们需要先明确 insert 语句的类型，自增锁的不同模式对不同 insert 类型的语句有不同的作用。

#### 插入类型
在 MySQL 中，插入类型包括“简单插入”、“批量插入”以及“混合模式插入”。

简单插入就是可以预先确定要插入的行数，主要包括单行和多行 insert，以及 replace 没有嵌套子查询的语句，但不包括 insert ... on duplicate key update。

批量插入就是事先不知道要插入的行数，这类插入通常涉及从其他表或外部数据源动态获取数据，或者在处理过程中逐步确定要插入的行数，主要包括 insert ... select、replace ... select、load data 等。

混合模式插入可以是简单插入语句，但是只需要为部分行指定自增值，比如下面的 insert：

```sql
insert into t(a, b) values(1, "1"), (null, "2"), (null, "3");
```

#### 自增锁模式
innodb_autoinc_lock_mode 可配置的自增锁模式包括：

+ 0 - Traditional 传统模式
+ 1 - Consecutive 连续模式
+ 2 - Interleaved 交错模式

##### 传统模式
传统模式更多的是为了兼容在引入 innodb_autoinc_lock_mode 变量之前的行为，并且解决“混合模式插入”的问题。

这种模式下，每当有插入操作发生时，都需要获取一个表级 AUTO-INC 锁，并在整个 insert 语句执行期间持有这个锁，注意和其他表级锁不一样，AUTO-INC 锁只需要在 insert 语句执行期间持有，而非在整个事务期间持有。

传统模式可以保证自增值是连续的，但是这种模式显然性能最差，会导致较高的锁竞争和等待时间。

##### 连续模式
这是 MySQL 5.1.22 到 8.0 之间的默认行为。

在连续模式下，对于“批量插入”，在整个 insert 语句期间依然使用 AUTO-INC 表级锁。

但对于“简单插入”，由于预先知道要插入的行数，所以只需要在一个轻量级锁（互斥锁）的控制下获得该 insert 语句需要的自增量即可释放轻量级锁，而非使用 AUTO-INC 表级锁。

当然，如果另一个事务，已经持有 AUTO-INC 表级锁，那么当前事务即使是“简单插入”，也只有等待 AUTO-INC 锁。

另外，连续模式在“混合模式插入”情况下，可能会出现丢号的情况。

举个例子：

在 innodb_autoinc_lock_mode 为 0 的情况下，执行如下 SQL：

```sql
CREATE TABLE `t` (
  `a` bigint NOT NULL AUTO_INCREMENT,
  `b` varchar(256) DEFAULT NULL,
  PRIMARY KEY (`a`)
) ENGINE=InnoDB AUTO_INCREMENT 100 DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci;

INSERT INTO t(a, b) VALUES(1, "1"), (null, "2"), (3, "3"), (null, "4");

INSERT INTO t(a, b) VALUES(null, "5");
```

##### 交错模式
这是 MySQL 8.0 及以后版本的默认行为。

交错模式中所有的 insert 语句都不使用 AUTO-INC 自增锁，而是基于轻量级锁在获取自增值之后即可释放，它允许更多的并发插入操作，这也是性能最高的锁模式。

连续模式在“混合模式插入”的情况下会出现丢号的情况，是因为基于轻量级锁获得该 insert 语句需要的自增量即可释放轻量级锁，而交错模式也是这样，所以交错模式下也会出现丢号的情况。

同时对于“批量插入”，传统模式和连续模式都会使用 AUTO-INC 表级锁在 insert 期间进行锁定，所以 insert 语句生成的自增值都是连续的，没有间隙。

而在交错模式下，如果同时并发执行 insert 语句，就可能会出现自增不连续。

#### 主从同步的影响
在基于 binlog 的主从同步场景下，对于交错模式并且 binlog 格式为 statement 的情况下，可能出现主从不一致问题。

举个例子。

存在表 t1，结构和数据如下：

```sql
CREATE TABLE `t1` (
  `a` bigint NOT NULL AUTO_INCREMENT,
  `b` varchar(255) DEFAULT NULL,
  `c` varchar(255) DEFAULT NULL,
  PRIMARY KEY (`a`)
) ENGINE=InnoDB AUTO_INCREMENT=5;

INSERT INTO `t1` (`a`, `b`, `c`) VALUES (1, '1', '1');
INSERT INTO `t1` (`a`, `b`, `c`) VALUES (2, '2', '2');
INSERT INTO `t1` (`a`, `b`, `c`) VALUES (3, '3', '3');
INSERT INTO `t1` (`a`, `b`, `c`) VALUES (4, '4', '4');
```

此时两个 session 并发执行下面的 SQL：

| Session A | Session B |
| --- | --- |
| | create table t2 like t1; |
| insert into t2 values(null, 5, 5); | insert into t2(c, d) select c, d from t1; |


首先 Session B 创建了与 t1 相同结构的 t2，然后两个 Session 同时并发插入数据。

在交错模式下，轻量级锁获取自增值之后就可以释放，而不必等到数据真正插入完毕。

那么就可能出现下面的情况：

1. Session B 先插入了两行记录 (1, 1, 1) 和 (2, 2, 2)
2. 然后 Session A 申请自增 id，得到 id = 3
3. 接着 Session B 继续插入，此时自增 id 来到 4，所以插入 (4, 3, 3) 和 (5, 4, 4)
4. 最后 Session A 开始实际插入 (3, 5, 5)

可以看到，Session B 的 insert 语句，生成的自增 id 并不连续。

当需要进行主从复制，并且 binlog 格式为 statement 时，binlog 会记录两个 Session 的 insert 语句，在从库上，要么先执行 Session A 的 insert，要么先执行 Session B 的 insert，但是无论什么情况，从库不会出现在主库上那样并发向 t2 插入数据的场景，所以生成的自增 id 是连续的。

而在连续模式或者传统模式下，对于 Session B 这种批量插入，会加表级 AUTO-INC 锁，所以不会出现这种情况。

所以，要解决这种主从不一致问题，一种是将自增锁模式切换为连续模式或者传统模式，另一种就是使用 row 或者 mix 作为 binlog 格式。

#### **如何优化自增锁**
如果自增锁成为了性能瓶颈，可以考虑以下策略：

+ 使用序列号生成器：在应用层生成主键值，避免数据库层面的自增锁。
+ 调整锁模式：在 MySQL 8.0 及更高版本中，通过设置参数 innodb_autoinc_lock_mode 来调整自增锁的行为，以适应不同的并发需求。

## 行级锁
MySQL 的行锁是在引擎层由各个引擎自己实现的。但并不是所有的引擎都支持行锁，比如 MyISAM 引擎就不支持行锁。不支持行锁意味着并发控制只能使用表锁，对于这种引擎的表，同一张表上任何时刻只能有一个更新在执行，这就会影响到业务并发度。

InnoDB 是支持行锁的，这也是 MyISAM 被 InnoDB 替代的重要原因之一。

### 两阶段锁协议
举个例子，下面的操作序列中，事务 B 的 update 语句执行时会是什么现象呢？假设字段 id 是表 t 的主键。

|  事务 A |   事务 B |
| --- | --- |
| begin;<br/>update t set k = k + 1 where id = 1;<br/>update t set k = k + 1 where id = 2; | |
|  | begin;<br/>update t set k = k + 2 where id = 1; |
| commit; |  |


这个问题的结论取决于事务 A 在执行完两条 update 语句后，持有哪些锁，以及在什么时候释放。实际上事务 B 的 update 语句会被阻塞，直到事务 A 执行 commit 之后，事务 B 才能继续执行。

实际上，事务 A 持有的两个记录的行锁，都是在 commit 的时候才释放的。

也就是说，在 InnoDB 事务中，行锁是在需要的时候才加上的，但并不是不需要了就立刻释放，而是要等到事务结束时才释放。这个就是「两阶段锁协议」。

「两阶段锁协议」对我们使用事务有什么帮助呢？如果你的事务中需要锁多个行，要把最可能造成锁冲突、最可能影响并发度的锁尽量往后放。

举个例子：假设要实现一个电影票在线交易业务，顾客 A 要在影院 B 购买电影票，这个业务涉及到以下操作：

1. 从顾客 A 账户余额中扣除电影票价；
2. 给影院 B 的账户余额增加这张电影票价；
3. 记录一条交易日志。

要完成这个交易，我们需要 update 两条记录，并 insert 一条记录。当然，为了保证交易的原子性，我们要把这三个操作放在一个事务中。那么，你会怎样安排这三个语句在事务中的顺序呢？

试想如果同时有另外一个顾客 C 要在影院 B 买票，那么这两个事务冲突的部分就是语句 2 了。因为它们要更新同一个影院账户的余额，需要修改同一行数据。

根据两阶段锁协议，不论你怎样安排语句顺序，所有的操作需要的行锁都是在事务提交的时候才释放的。所以，如果你把语句 2 安排在最后，比如按照 3、1、2 这样的顺序，那么影院账户余额这一行的锁时间就最少，这就最大程度地减少了事务之间的锁等待，提升了并发度。

### 共享锁和排他锁
InnoDB 实现了以下两种类型的行锁：

+ 共享锁（S）：允许一个事务去读一行，阻止其他事务获得相同数据集的排它锁，多个事务可以同时拥有同一行的共享锁。
+ 排他锁（X）：允许获取排他锁的事务更新数据，阻止其他事务获得相同数据集的共享锁和排他锁，每次只有一个事务可以获得排他锁。

两种行锁的兼容情况如下：

| 当前锁类型\请求锁类型 | S（共享锁） | X（排他锁） |
| --- | --- | --- |
| S（共享锁） | 兼容 | 冲突 |
| X（排他锁） | 冲突 | 冲突 |


常见的 SQL 语句，在执行时，所加的行锁如下：

| SQL | 行锁类型 | 说明 |
| --- | --- | --- |
| insert … | 排他锁 | 自动加锁 |
| update … | 排他锁 | 自动加锁 |
| delete … | 排他锁 | 自动加锁 |
| select … | 不加锁 |  |
| select ... lock in share mode | 共享锁 | 需要手动在 select 之后加 lock in share mode |
| select ... for update | 排他锁 | 需要手动在 select 之后加 for update |


普通的 select 语句是不会对记录加锁的，因为它属于快照读。如果要在查询时对记录加行锁，可以使用 for update 或者 lock in share mode，这种查询会加锁的语句称为 **当前读**。

行级锁的类型主要有三类：

+ Record Lock，行锁，仅仅把一条行记录锁上；
+ Gap Lock，间隙锁，锁定一个范围，不包含记录本身；
+ Next-Key Lock：Record Lock + Gap Lock 的组合，锁定一个范围，并且锁定记录本身。

### Record Lock
Record Lock 称为行锁/记录锁，锁住的是一条记录，并且行锁是有 S 锁和 X 锁之分的：

+ 当一个事务对一条记录加了 S 型行锁后，其他事务也可以继续对该记录加 S 型行锁（S 型与 S 锁兼容），但是不可以对该记录加 X 型行锁（S 型与 X 锁不兼容）
+ 当一个事务对一条记录加了 X 型行锁后，其他事务既不可以对该记录加 S 型行锁（S 型与 X 锁不兼容），也不可以对该记录加 X 型行锁（X 型与 X 锁不兼容）。

举个例子，当一个事务执行了下面这条语句：

```sql
begin;
select * from t where id = 34 for update;
```

就是对 t 表中主键 id 为 34 的这条记录加上 X 型的行锁，这样其他事务就无法删除或更新这条记录。

![image-20240918213008569](/img/image-20240918213008569.png)

注意：

1. 这里最初加的是临键锁，只不过因为这个查询是唯一索引的等值查询，所以临键锁优化为行锁。
2. 其他事务插入一条 id = 34 的新记录并不会被阻塞，而是会报主键冲突的错误，主键有唯一性约束。

### Gap Lock
Gap Lock 称为间隙锁，只存在于 **可重复读隔离级别**，目的是为了解决可重复读隔离级别下幻读的现象。

假设，表中有一个范围 id 为 (18, 29) 间隙锁，那么其他事务就无法插入 id = 19、20 …… 28 这些记录了，这样就有效的防止了幻读现象的发生。

![image-20240918213015911](/img/image-20240918213015911.png)

间隙锁虽然存在 X 型间隙锁和 S 型间隙锁，但是 **间隙锁之间是兼容的，即两个事务可以同时持有包含共同间隙范围的间隙锁，并不存在互斥关系，因为间隙锁的目的是防止插入幻影记录而提出的**。

两个事务可以包含共同间隙范围的间隙锁，间隙范围有两个含义：

+ 其一是两个间隙锁的间隙区间完全一样
+ 其二是一个间隙锁包含的间隙区间是另一个间隙锁包含间隙区间的子集

### Next-Key Lock
Next-Key Lock 称为临键锁，是 Record Lock + Gap Lock 的组合，锁定一个范围以及记录本身。

默认情况下，加锁的基本单位就是临键锁。

假设，表中有一个范围为 (16, 18] 的 Next-Key Lock，那么其他事务即不能插入 id = 17 记录，也不能修改 id = 18 这条记录。

![image-20240918213025477](/img/image-20240918213025477.png)

所以，Next-Key Lock 既能保护该记录，又能阻止其他事务将新记录插入到被保护记录前面的间隙中。

**Next-Key Lock 是包含间隙锁 + 行锁的，如果一个事务获取了 X 型的 Next-Key Lock，那么另外一个事务在获取相同范围的 X 型的 Next-Key Lock 时会被阻塞**。

比如，一个事务持有了范围为 (1, 10] 的 X 型的 Next-Key Lock，那么另外一个事务在获取相同范围的 X 型的 Next-Key Lock 时，就会被阻塞。

虽然相同范围的间隙锁是多个事务相互兼容的，但对于行锁，还要考虑 X 型与 S 型的关系，X 型的行锁与 X 型的行锁是冲突的。

但是，注意一种特殊的情况，比如多个事务同时对 (100, +∞] 范围加临键锁，这里其实也是不会冲突的，多个事务可以同时持有，因为 +∞ 并不是一个真实的记录，自然也无需考虑 X 型和 S 型的关系。

### 插入意向锁
一个事务在插入一条记录时，需要判断插入位置是否已被其他事务加了间隙锁（临键锁也包含间隙锁）。

如果有的话，插入操作就会发生 **阻塞**，直到拥有间隙锁的事务提交为止（释放间隙锁的时刻），在此期间会生成一个 **插入意向锁**，表明有事务想在某个区间插入新记录，但是现在处于等待状态。

插入意向锁的生成时机：每插入一条新记录，都需要看一下待插入记录的下一条记录上是否已经被加了间隙锁，如果已加上了间隙锁，此时会生成一个插入意向锁，然后锁的状态设置为等待状态（MySQL 加锁时，是先生成锁结构，然后设置锁的状态，如果锁状态是等待状态，并不是意味着事务成功获取到了锁，只有当锁状态为正常状态时，才代表事务成功获取到了锁，现象就是 insert 语句会被阻塞。

举个例子，假设事务 A 已经对表加了一个范围为 (18, 29) 间隙锁。

![image-20240918213030717](/img/image-20240918213030717.png)

当事务 A 还没提交时，事务 B 向该表插入一条 id = 19 的新记录，这时会判断到插入的位置已经被事务 A 加了间隙锁，于是事务 B 会生成一个插入意向锁，然后将锁的状态设置为等待状态（MySQL 加锁时，会先生成锁结构，然后设置锁的状态，如果锁状态是等待状态，并不意味着事务成功获取到了锁，只有当锁状态为正常状态时，才代表事务成功获取到了锁），此时事务 B 就会发生阻塞，直到事务 A 提交了事务。

插入意向锁名字虽然有意向锁，但是它并 **不是意向锁，它是一种特殊的间隙锁，属于行级别锁**。

如果说间隙锁锁住的是一个区间，那么插入意向锁锁住的就是一个点，因而从这个角度来说，插入意向锁确实是一种特殊的间隙锁。

## 如何加锁
由于 InnoDB 的行锁加锁规则比较复杂，所以我们详细聊聊。

### 加锁规则
四条规则：

1. 加锁的对象是 **索引**
2. 查找过程中访问到的索引对象才会加锁
3. 加锁的基本单位是 Next-Key Lock 临键锁，是一个（前开后闭」的区间
4. Next-Key Lock 临键锁在一些场景下会优化成行锁或间隙锁

临键锁优化的情况：

1. 唯一索引等值查询：
    - 查询的记录存在，在索引树上定位记录后，对该记录的索引加上的临键锁优化为「行锁」。
    - 当查询的记录不存在，在索引树找到第一条大于该查询记录的记录后，将该记录的索引中临键锁优化 为「间隙锁」。
2. 唯一索引范围查询：当唯一索引进行范围查询时，会对每一个扫描的索引加临键锁，遇到下面这些情况，会优化成行锁或者间隙锁
    - 针对「大于等于」的范围查询，因为存在等值查询的条件，如果等值查询的记录存在表中，那么该记录的索引中的临键锁会优化成行锁。
    - 针对「小于或者小于等于」的范围查询，要看条件值的记录是否存在于表中： 
        * 当条件值的记录不在表中，则不管是小于还是小于等于条件的范围查询，扫描到终止范围查询的记录时，该记录的索引的临键锁会优化成间隙锁，其他扫描到的记录，都是在这些记录的索引上加临键锁。
        * 当条件值的记录在表中，
            + 如果是小于条件的范围查询，扫描到终止范围查询的记录时，该记录的索引的临键锁会优化成间隙锁，其他扫描到的记录，都是在这些记录的索引上加临键锁；
            + 如果小于等于条件的范围查询，扫描到终止范围查询的记录时，该记录的索引临键锁不会优化成间隙锁。其他扫描到的记录，都是在这些记录的索引上加临键锁。
3. 非唯一索引等值查询：当使用非唯一索引进行等值查询时，因为存在两个索引，一个是主键索引，一个是非唯一索引（二级索引），所以在加锁时，如果通过回表访问了主键索引，那么也会对主键索引加行锁。 
    - 当查询的记录存在时，由于不是唯一索引，可能还存在索引值相同的记录，所以非唯一索引等值查询的过程是一个扫描的过程，直到扫描到第一个不符合条件的二级索引记录才停止扫描，在扫描的过程中，对扫描到的二级索引记录加的是临键锁，而对于第一个不符合条件的二级索引记录，该二级索引的临键锁会优化为间隙锁。
    - 当查询的记录不存在时，扫描到第一条不符合条件的二级索引记录，该二级索引的临键锁会优化成间隙锁，由于没有记录存在，所以不可能访问到主键索引，所以也不会对主键索引加行锁。
4. 非唯一索引范围查询，索引的临键锁不会有优化为间隙锁和行锁的情况，也就是非唯一索引进行范围查询时，对二级索引记录加锁都是加临键锁。

**如果查询条件没有使用索引**

如果当前读的查询语句，没有使用索引列作为查询条件，或者查询语句没有走索引查询，导致全表扫描。那么，每一条记录的索引上都会加临键锁，相当于锁住了全表，此时如果其他事务对该表进行增、删、改操作的时候，都会被阻塞。

不只是当前读查询语句不加索引才会导致这种情况，update 和 delete 语句如果查询条件不加索引，那么由于扫描的方式是全表扫描，于是就会对每一条记录的主键索引上都会加临键锁，这样就相当于锁住的全表。

因此，在执行 update、delete、select ... for update 等具有加锁性质的语句，一定要检查语句是否走了索引，如果是全表扫描的话，会对每一个索引加临键锁，相当于把整个表锁住了，这是挺严重的问题。

但是，这种在非索引列上的当前读查询并不是行锁升级为表锁，只是因为扫描的方式是全表扫描，于是对每一条记录的主键索引都加上临键锁，相当于锁住全表。

### 查看加锁情况
如何查看意向锁和行级锁的加锁情况：

```sql
# 详细信息
select * from performance_schema.data_locks;

# 简略信息
select object_schema,object_name,index_name,lock_type,lock_mode,lock_data from performance_schema.data_locks;
```

常见字段含义：

1. lock_type 值的类型：
    1. TABLE：表锁
    2. RECORD：行锁
2. lock_mode 值的类型：
    1. 表锁：
        1. `IX`：意向独占锁
        2. `IS`：意向共享锁
    2. 行锁：
        1. `X`：独占锁，临键锁
        2. `S`：共享锁，临键锁
        3. `X,REC_NOT_GAP`：独占锁，行锁
        4. `S,REC_NOT_GAP`：共享锁，行锁
        5. `X,GAP`：独占锁，间隙锁
        6. `S,GAP`：共享锁，间隙锁
3. lock_data：
    1. lock_data 值为加锁范围的右边界，左边界为表中 lock_data 值的上一行的记录值。
    2. 对于二级索引来说，lock_data 有两部分值，索引以及对应的主键值。

### 唯一索引等值查询
#### 查询的记录存在
查询的记录存在，在索引树上定位记录后，对该记录的索引加的临键锁会优化为「行锁」。

```sql
begin;
select * from t_user where id = 5 for update;
```

![image-20240918213044544](/img/image-20240918213044544.png)

对 id = 5 的记录加了临键锁 (1, 5]，但是由于是唯一索引（主键索引是唯一索引），所以临键锁优化为行锁。

#### 查询的记录不存在
当查询的记录不存在，在索引树找到第一条大于该查询记录的记录后，将该记录的索引中临键锁优化为「间隙锁」。

```sql
begin;
select * from t_user where id = 7 for update;
```

![image-20240918213050946](/img/image-20240918213050946.png)

在索引树上找到第一条 id > 7 的记录即 id = 10 的记录，对 id = 10 的记录加的临键锁 (5, 10] 被优化为间隙锁 (5, 10)。

### 唯一索引范围查询
#### 针对大于等于的范围查询
针对「大于等于」的范围查询，因为存在等值查询的条件，如果等值查询的记录存在表中，那么该记录的索引中的临键锁会优化成行锁。

所以有三种情况：

+ 范围查询是大于
+ 范围查询是大于等于，但等值查询的记录不存在表中
+ 范围查询是大于等于，并且等值查询的记录存在表中

1）范围查询是大于

```sql
begin;
select * from t_user where id > 10 for update;
```

![image-20240918213055853](/img/image-20240918213055853.png)

1. 扫描的第一行是 id = 15，由于查询该记录不是等值查询（不是大于等于条件查询），所以对该主键索引加的是 (10, 15] 的临键锁。
2. 由于是范围查询，就会继续向后扫描，所以还会加 (15, 20]、(20, +∞] 的临键锁。
3. 这里注意，虽然表中最后一条记录是 id = 20 的记录，但是实际在 InnoDB 存储引擎中，会用一个特殊的记录来标识最后一条记录，该特殊的记录的名字是 supremum pseudo-record，所以对该主键索引加的是范围为 (20，+∞] 的临键锁。

2）范围查询是大于等于，但等值查询的记录不存在表中

```sql
begin;
select * from t_user where id >= 9 for update;
```

![image-20240918213100593](/img/image-20240918213100593.png)

可以看到，这时对扫描的记录 id = 10、15、20、+∞ 都加的是临键锁，没有任何优化。

3）范围查询是大于等于，并且等值查询的记录存在表中

```sql
begin;
select * from t_user where id >= 10 for update;
```

![image-20240918213105137](/img/image-20240918213105137.png)

针对 **大于等于** 的范围查询，因为存在等值查询的条件，并且等值查询的记录 id = 10 是存在表中的，那么该记录的索引中临键锁会 **优化成行锁**。

所以上面的 id = 10 的记录加的锁是行锁。

#### 针对小于或者小于等于的范围查询
针对「小于或者小于等于」的范围查询，要看条件值的记录是否存在于表中。

**条件值的记录不在表中**

当条件值的记录不在表中，则不管是小于还是小于等于条件的范围查询，扫描到终止范围查询的记录时，该记录的索引的临键锁会优化成间隙锁，其他扫描到的记录，都是在这些记录的索引上加临键锁。

```sql
begin;
select * from t_user where id < 16 for update;
```

![image-20240918213109162](/img/image-20240918213109162.png)

```sql
begin;
select * from t_user where id <= 16 for update;
```

![image-20240918213115026](/img/image-20240918213115026.png)

**条件值的记录在表中**

+ 如果是小于条件的范围查询，扫描到终止范围查询的记录时，该记录的索引的临键锁会优化成间隙锁，其他扫描到的记录，都是在这些记录的索引上加临键锁；
+ 如果小于等于条件的范围查询，扫描到终止范围查询的记录时，该记录的索引临键锁不会优化成间隙锁。其他扫描到的记录，都是在这些记录的索引上加临键锁。

```sql
begin;
select * from t_user where id < 20 for update;
```

![image-20240918213119030](/img/image-20240918213119030.png)

```sql
begin;
select * from t_user where id <= 20 for update;
```

### 非唯一索引等值查询
当使用非唯一索引进行等值查询时，因为存在两个索引，一个是主键索引，一个是非唯一索引（二级索引），所以在加锁时，如果通过回表访问了主键索引，那么也会对主键索引加行锁。

这里正是加锁规则的第 2 点：查找过程中访问到的索引对象才会加锁。

注意 lock in share mode 和 for update 的区分：

+ 通过 lock in share mode 查询，如果使用了覆盖索引，那么存储引擎就不会访问主键索引。
+ 通过 for update 查询，即使使用了覆盖索引，存储引擎还是会访问主键索引。

#### 查询的记录存在
当查询的记录存在时，由于不是唯一索引，可能还存在索引值相同的记录，所以非唯一索引等值查询的过程是一个扫描的过程，直到扫描到第一个不符合条件的二级索引记录才停止扫描，在扫描的过程中，对扫描到的二级索引记录加的是临键锁，而对于第一个不符合条件的二级索引记录，该二级索引的临键锁会优化为间隙锁。

```sql
begin;
select * from t_user where age = 21;
```

![image-20240918213127784](/img/image-20240918213127784.png)

注意 lock_data 的值：左边是二级索引的值，右边是其对应主键索引的值。

+ 对 age = 21 的二级索引加的是临键锁 (20, 21]
+ 对于扫描过程中，第一个不符合条件的二级索引记录，该二级索引的临键锁会 **优化成间隙锁**，所以对 age = 22 的二级索引加的是间隙锁 (21, 22)
+ 查询过程中出现了回表，访问了主键索引，所以对 id = 5 的主键索引加了行锁 [5]。

当查询的记录不存在时，扫描到第一条不符合条件的二级索引记录，该二级索引的临键锁会优化成间隙锁，由于没有记录存在，所以不可能访问到主键索引，所以也不会对主键索引加行锁。

```sql
begin;
select id from t_user where age = 21 lock in share mode;
```

![image-20240918213131702](/img/image-20240918213131702.png)

+ 通过 lock in share mode 查询，如果使用了覆盖索引，那么存储引擎就不会访问主键索引，所以这里没有 id = 5 的主键索引的行锁。

#### 查询的记录不存在
当查询的记录不存在时，**扫描到第一条不符合条件的二级索引记录，该二级索引的临键锁会优化成间隙锁，由于没有记录存在，所以不可能访问到主键索引，所以也不会对主键索引加行锁。**

```sql
begin;
select * from t_user where age = 23 for update;
```

![image-20240918213136130](/img/image-20240918213136130.png)

### 非唯一索引范围查询
非唯一索引范围查询，索引的临键锁不会有优化为间隙锁和行锁的情况，也就是非唯一索引进行范围查询时，对二级索引记录加锁都是加临键锁。

如果通过二级索引访问了主键索引，还要在主键索引上加行锁。

```sql
begin;
select * from t_user where age > 21 for update;
```

![image-20240918213140425](/img/image-20240918213140425.png)

### 当前读的查询没有使用索引
如果当前读的查询语句，没有使用索引列作为查询条件，或者查询语句没有走索引查询，导致扫描是全表扫描。那么，每一条记录的索引上都会加临键锁，相当于锁住了全表，此时如果其他事务对该表进行增、删、改操作的时候，都会被阻塞。

```sql
begin;
select * from t_user where name = 'name' for update;
```

![image-20240918213144838](/img/image-20240918213144838.png)

## 加锁例子
再来看一些加锁例子

表结构如下：

```sql
CREATE TABLE `t`  (
  `id` int NOT NULL,
  `c` int NULL DEFAULT NULL,
  `d` int NULL DEFAULT NULL,
  PRIMARY KEY (`id`) USING BTREE,
  INDEX `c`(`c` ASC) USING BTREE
) ENGINE = InnoDB CHARACTER SET = utf8mb4 COLLATE = utf8mb4_0900_ai_ci ROW_FORMAT = Dynamic;

INSERT INTO `t` VALUES (0, 0, 0);
INSERT INTO `t` VALUES (5, 5, 5);
INSERT INTO `t` VALUES (10, 10, 10);
INSERT INTO `t` VALUES (15, 15, 15);
INSERT INTO `t` VALUES (20, 20, 20);
INSERT INTO `t` VALUES (25, 25, 25);
```

### Ex1
等值查询间隙锁

```sql
begin;
update t set d = d + 1 where id = 7;
```

首先这是通过主键查询，并且主键 id = 7 不在表记录中，所以在聚簇索引树上找到第一条 id > 7 的记录，即 id = 10 的记录，将 id = 10 的临键锁优化为间隙锁。

![image-20240918213148657](/img/image-20240918213148657.png)

### Ex2
非唯一索引等值锁

```sql
begin;
select id from t where c = 5 lock in share mode;
```

首先这是二级索引等值查询，c = 5 存在表中，但是还会向右扫描，直到 c ≠ 5，即 c = 10 为止，所以对 c = 5 加的是临键锁，对 c = 10 的记录加的是间隙锁。

其次，使用的是 lock in share mode，查询的是 id，不会回表，所以无需查聚簇索引树。

![image-20240918213207053](/img/image-20240918213207053.png)

### Ex3
<font style="color:rgb(59, 67, 81);">主键索引范围锁</font>

```sql
begin;
select * from t where id >= 10 and id < 11 for update;
```

这个 SQL 有两段唯一索引的范围查询：

1. 大于等于类型的唯一索引范围查询，并且等值条件 id = 10 存在表中，所以将 id = 10 的临键锁优化为行锁。
2. 小于类型的唯一索引范围查询，条件值不在表中，所以对扫描到的终止范围的记录 id = 15 加的临键锁会优化为间隙锁。

![image-20240918213211934](/img/image-20240918213211934.png)

### Ex4
非唯一索引范围锁

```sql
begin;
select * from t where c >= 10 and c < 11 for update;
```

对于非唯一索引的范围查询，不会有任何优化为间隙锁和行锁的情况。

注意，这里实际访问的二级索引有 c = 10、15，所以会对 c = 10、15 的二级索引记录加临键锁，并且查询的字段是`*`，所以对于 c = 10 的记录会有回表情况，所以还会对 id = 10 的主键索引记录加行锁。

![image-20240918213220975](/img/image-20240918213220975.png)

### Ex5
```sql
begin;
select * from t where id > 10 and id <= 15 for update;
```

这个 SQL 有两段唯一索引的范围查询：

1. 大于类型的唯一索引范围查询，第一条访问的主键索引是 id = 15 的记录，这种类型不会有任何优化，所以对 id = 15 的记录加的是临键锁。
2. 小于等于类型的唯一索引范围查询，条件值在表中，所以对扫描到的终止范围的记录 id = 15 加的临键锁，也不会有优化。

![image-20240918213225227](/img/image-20240918213225227.png)

### Ex6
非唯一索引存在等值

```sql
insert into t values(30,10,30);

begin;
delete from t where c = 10;
```

当插入了这条记录之后，表中 c = 10 的记录就有两条了。

删除语句是一个非唯一索引的等值查询，并且查询条件存在表中，所以会进行扫描，直到第一个不符合条件的二级索引记录才停止，在扫描的过程中，对扫描到的二级索引记录加的是临键锁，而对于第一个不符合条件的二级索引记录，该二级索引的临键锁会优化为间隙锁。

最终加的锁有：id = 10，c = 10 的临键锁，id = 30，c = 10 的临键锁，c = 15 的间隙锁，同时访问了 id = 10、30 的主键索引，所以还会对 id = 10、30 的主键索引加行锁。

![image-20240918213228981](/img/image-20240918213228981.png)

### Ex7
Limit 加锁

```sql
begin;
delete from t where c = 10 limit 2;
```

表 t 里 c = 10 的记录只有两条，因此加不加 limit 2，删除的效果都是一样的，但是加锁的效果却不同。

这里的 delete 语句明确加了 limit 2 的限制，因此在遍历到 c = 10，id = 30 这一行之后，满足条件的语句已经有两条，循环就结束了，不会访问后面 c = 15 的记录。

![image-20240918213233769](/img/image-20240918213233769.png)

## 间隙锁引起的死锁
> 存储引擎 InnoDB，隔离级别可重复读（RR）。
>

最后再来看两个死锁的例子。

### Ex1
表结构：

```sql
CREATE TABLE `t_order`  (
  `id` int NOT NULL AUTO_INCREMENT,
  `order_no` int NULL DEFAULT NULL,
  `create_date` datetime NULL DEFAULT NULL,
  PRIMARY KEY (`id`) USING BTREE,
  INDEX `idx_order`(`order_no` ASC) USING BTREE
) ENGINE = InnoDB AUTO_INCREMENT = 7 CHARACTER SET = utf8mb4 COLLATE = utf8mb4_0900_ai_ci ROW_FORMAT = Dynamic;

INSERT INTO `t_order` VALUES (1, 1001, '2024-08-02 13:59:07');
INSERT INTO `t_order` VALUES (2, 1002, '2024-08-02 13:59:14');
INSERT INTO `t_order` VALUES (3, 1003, '2024-08-02 13:59:21');
INSERT INTO `t_order` VALUES (4, 1004, '2024-08-02 13:59:30');
INSERT INTO `t_order` VALUES (5, 1005, '2024-08-02 13:59:36');
INSERT INTO `t_order` VALUES (6, 1006, '2024-08-02 13:59:59');
```

假设此时有两个事务，一个事务要插入订单 1007 ，另外一个事务要插入订单 1008，因为需要对订单做幂等性校验，所以两个事务都需要先查询该订单是否存在，不存在才插入记录，过程如下：

|  事务 A | 事务 B  |
| --- | --- |
| begin; | begin; |
| 检查 1007 订单是否存在<br/>select id from t_order where order_no = 1007 for update; |  |
|  | 检查 1008 订单是否存在<br/>select id from t_order where order_no = 1008 for update; |
| 如果没有，则插入订单记录<br/>insert into t_order(order_no, create_date) values(1007, now());<br/>阻塞等待 …… |  |
|  | 如果没有，则插入订单记录<br/>insert into t_order(order_no, create_date) values(1008, now());<br/>阻塞等待 …… |


可以看到，两个事务都陷入了等待状态（前提没有打开死锁检测），也就是发生了死锁，因为都在相互等待对方释放锁。

原因是：两个事务在执行检查订单是否存在的查询 SQL 之后，都在 (1006, +∞] 加了临键锁，如下：

![image-20240918213239498](/img/image-20240918213239498.png)

这里由于 +∞ 不是真实的记录，所以两个事务可以同时持有 (1006, +∞] 范围的临键锁。

接下来事务 A 的插入操作，发现事务 B 在 (1006, +∞] 间隙内存在间隙锁，所以事务 A 会插入一个「插入意向锁 INSERT_INTENTION」，然后进入阻塞，如下：

![image-20240918213243410](/img/image-20240918213243410.png)

同理，当事务 B 执行插入语句时，发现该间隙已经被事务 A 插入了间隙锁，所以会向该间隙内插入一个插入意向锁，然后进入阻塞。

两个事务都在等待对方事务的间隙锁释放，于是就造成了循环等待，导致死锁。

#### 为什么使用 for update
这里在查询记录是否存在的时候，使用了 select ... for update 语句，目的为了防止事务执行的过程中，有其他事务插入了记录，而出现幻读的问题。

如果没有使用 select ... for update 语句，而使用了单纯的 select 语句，如果是两个订单号一样的请求同时进来，就会出现两个重复的订单，有可能出现幻读，如下：

|  事务 A | 事务 B  |
| --- | --- |
| begin; | begin; |
| 检查 1007 订单是否存在<br/>select id from t_order where order_no = 1007 |  |
|  | 检查 1007 订单是否存在<br/>select id from t_order where order_no = 1007 |
| 如果没有，则插入订单记录<br/>insert into t_order(order_no, create_date) values(1007, now()); |  |
|  | 如果没有，则插入订单记录<br/>insert into t_order(order_no, create_date) values(1007, now()); |
| 存在两条订单为 1007 的记录（幻读） | |


#### 如何避免死锁
死锁的四个必要条件是互斥、占有且等待、不可强占用、循环等待。只要系统发生死锁，这些条件必然成立，所以只要破坏任意一个条件就死锁就不会成立。

在数据库层面，有两种策略通过「打破循环等待条件」来解除死锁状态： 

1. 设置事务等待锁的超时时间。当一个事务的等待时间超过该值后，就对该事务进行回滚，于是锁就释放了，另一个事务就可以继续执行了。在 InnoDB 中，参数 innodb_lock_wait_timeout 是用来设置超时时间的，默认值时 50 秒。
2. 开启主动死锁检测。主动死锁检测在发现死锁之后，主动回滚死锁链条中的某个事务，让其他事务得以继续执行。在 InnoDB 中，参数 innodb_deadlock_detect 设置为 on 表示开启死锁检测，默认是开启的。

上面这个两种策略是「当有死锁发生时」的避免方式。

我们可以回归业务的角度来预防死锁，对订单做幂等性校验的目的是为了保证不会出现重复的订单，那我们可以直接将 order_no 字段设置为唯一索引列，利用它的唯一性来保证订单表不会出现重复的订单，不过有一点不好的地方就是在我们插入一个已经存在的订单记录时就会抛出异常。

数据表如下：

当事务 B 的 update 语句执行之后，整个表加的锁如下：

![image-20240918213248990](/img/image-20240918213248990.png)

事务 A 和事务 B 分别加了 (20, 30) 的间隙锁。

当事务 A 执行 insert 语句之后，整个表加的锁如下：

![image-20240918213253241](/img/image-20240918213253241.png)

可以看到由于事务 A 需要向事务 B 生成的间隙锁（范围 (20, 30)）中插入一条记录，所以事务 A 的插入操作生成了一个插入意向锁 INSERT_INTENTION，insert 语句会被阻塞。

事务 B 插入时同理，由于事务 B 需要向事务 A 生成的间隙锁（范围 (20, 30)）中插入一条记录，所以事务 B 的插入操作生成了一个插入意向锁 INSERT_INTENTION，insert 语句会被阻塞。

所以，事务 A 和事务 B 在执行完后 update 语句后都持有范围为 (20, 30) 的间隙锁，而接下来的插入操作为了获取到插入意向锁，都在等待对方事务的间隙锁释放，于是就造成了循环等待，满足了死锁的四个条件：互斥、占有且等待、不可强占用、循环等待，因此发生了死锁。
