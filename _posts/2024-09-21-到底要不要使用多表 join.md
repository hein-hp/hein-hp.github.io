## 前言

在 MySQL 中，join 就是将两张以上的表的行根据 on 后的条件结合起来，组成了一张临时表，对这张临时表可以进行在单表上进行的操作，如 where 过滤、order by 排序、group by 分组、limit 分页等。

两张或多张表能够进行 join，那么它们至少要有一个公共的 field，它强调实际含义，而非具体的名称，比如用户表和订单表，在订单表中存储 user_id，就可以和用户表的 id 作为公共的 field，它们名称不同，但是实际含义是一致的。

这里我们不对 join 如何使用做说明，而是想聊一聊 join 有哪些实现方式，以及一些优化点，主要参考《MySQL 45 讲》。

## join 是如何执行的
实际上，在 MySQL 的版本历史中，一共有三种 join 算法：

1. 嵌套循环连接算法 Nest Loop Join，NLJ
2. 块嵌套循环连接算法 Block Nested Loop，BNL
3. 哈希连接 Hash Join（MySQL 8.0.18 之后启用）

这里我们就先来看看在 MySQL 8.0.18 之前使用的两种嵌套循环算法，为了便于我们分析，先创建一个示例表。

```sql
CREATE TABLE `t2` (
  `id` int(11) NOT NULL,
  `a` int(11) DEFAULT NULL,
  `b` int(11) DEFAULT NULL,
  PRIMARY KEY (`id`),
  KEY `a` (`a`)
) ENGINE=InnoDB;

delimiter ;;
create procedure idata()
begin
  declare i int;
  set i = 1;
  while(i <= 1000)do
    insert into t2 values(i, i, i);
    set i = i + 1;
  end while;
end;;
delimiter ;
call idata();
 
create table t1 like t2;
insert into t1 (select * from t2 where id <= 100)
```

这两个表都有一个主键索引 id 和一个索引 a，字段 b 上无索引。存储过程 idata() 往表 t2 里插入了 1000 行数据，在表 t1 里插入 100 行数据。

### NLJ
#### 无索引等值连接
假设有下面的一条 SQL，你会如何执行？

```sql
select
  *
from t1 straight_join t2 on t1.b = t2.b;
```

我们先解释一下，straight_join 是 MySQL 中的一个关键字，它用于强制查询优化器按照指定的表顺序来连接（join）表。

这里 straight_join 强制 MySQL 先访问 t1 表，然后用 t1 的数据去匹配 t2 表中的记录，t1 作为驱动表，而 t2 作为被驱动表。

如果我们直接使用 join 进行连接，优化器可能会选择 t1 或者 t2 作为驱动表，这样可能会影响我们分析 SQL 的执行流程，所以使用 straight_join 指定固定的连接方式进行查询。

我们来考虑应该如何执行这个 SQL，可以基于一个简单的嵌套循环：

```java
for each row in t1 all {
    for each row in t2 all {
        if row match, send to client;
    }
}
```

由于表 t2 的 b 字段上不存在索引，因此这个 SQL 的执行流程是这样的：

1. 从表 t1 中读入一行数据 row；
2. 从数据行 row 中，取出 a 字段到表 t2 里去查找（进行全表扫描，可以理解为 O(n) 的复杂度）；
3. 取出表 t2 中满足条件的行，跟 row 组成一行，作为结果集的一部分；
4. 重复执行步骤 1 到 3，直到表 t1 的末尾循环结束。

这样看来，这个 SQL 将会扫描表 t2 将近 100 次，总共扫描 100 * 1000 = 10 万行数据。

这还只是两个小表，如果 t1 和 t2 都是千万级别的大表呢，那么扫描次数都要达到万亿级别了，这显然是不可取的，我们看一下执行计划，重点关注 Extra 字段。

可以看出，MySQL 也并没有使用 NLJ 算法来实现这种无法使用索引的等值连接，而是使用了我们接下来要说的 BNL 算法。

#### 索引等值连接
假如 join 连接可以使用索引呢，参考下面的 SQL：

```sql
select
  *
from t1 straight_join t2 on t1.a = t2.a;
```

我们先看一下执行计划。

![image-20240921193715294](/img/image-20240921193715294.png)

当然，这还是基于一个简单的嵌套循环

```java
for each row in t1 all {
    for each row in t2 ref {
        if row match, send to client;
    }
}
```

但是这里对于被驱动表 t2 的处理是不同的，由于 t2 的字段 a 上有索引，所以在 join 过程会使用这个索引，因此这个 SQL 的执行流程是这样的：

1. 从表 t1 中读入一行数据 row；
2. 从数据行 row 中，取出 a 字段到表 t2 里去查找（在二级索引 B+ 树上查找，可以理解为 O(logn) 的复杂度）；
3. 取出表 t2 中满足条件的行，跟 row 组成一行，作为结果集的一部分；
4. 重复执行步骤 1 到 3，直到表 t1 的末尾循环结束。

在这个流程中，对驱动表 t1 做了全表扫描，这个过程需要扫描 100 行，而对于每一行 row，根据 a 字段去表 t2 查找，走的是 B+ 树搜索过程。由于我们构造的数据都是一一对应的，因此每次的搜索过程都只扫描一行，也是总共扫描 100 行。所以，整个执行流程，总扫描行数是 200。

注意，因为 t2 表的 a 字段存在索引，所有只是在二级索引 B+ 树上搜索，而不会实际扫描行记录，所以这里 t2 表扫描不是 1000 行，而是 100 行（等值匹配）。

### BNL
我们在上面的一条执行计划中也看出了，当无法使用索引的等值连接时，就会使用 BNL 算法。

其实从本质上来说，BNL 也是一种循环嵌套算法，只不过该算法将外部循环中读取的行缓存在内存中来减少读取内部循环表的次数。

还是上面的 SQL，如下：

```sql
select
  *
from t1 straight_join t2 on t1.b = t2.b;
```

由于被驱动表 t2 上 b 字段无索引，所以使用 BNL 算法，流程如下：

1. 首先将连接需要的 t1 表的数据读取连接缓冲区 join_buffer（意味着后续无需再次从磁盘中读取），这里由于 select *，所以将整个表 t1 放入 join_buffer。
2. 然后循环遍历表 t2 的记录 row，将 row 中的 b 字段和 join_buffer 中的数据比对，满足 join 条件的加入结果集。

整个过程，对表 t1 和表 t2 都仅做了一次全表扫描，因此总的扫描次数是 1100，由于 join_buffer 是以无序数组的方式组织的，因此对表 t2 中的每一行，都要做 100 次判断，总共需要在内存中做的判断次数是：100 * 1000 = 10 万次。

但这还不是 BNL 名称的由来。

我们考虑这样一种情况，假设表 t1 数据量太大，join_buffer 存不下怎么办？

join_buffer 的大小是由参数 join_buffer_size 设定的，默认值是 256k，如果放不下表 t1 的所有数据，就会进行分块 join，执行过程就变成了：

1. 扫描表 t1，顺序读取数据行放入 join_buffer 中直到 join_buffer 满，继续第 2 步；
2. 循环遍历表 t2 的记录 row，将 row 中的 b 字段和 join_buffer 中的数据比对，满足 join 条件的加入结果集。
3. 清空 join_buffer；
4. 继续扫描表 t1，继续读取数据行放入 join_buffer 中，继续执行第 2 步。

这个流程才体现出了这个算法名字中 Block 的由来，表示分块 join。

可以看到，这时候由于表 t1 可能被分为多次放入 join_buffer 中，导致表 t2 会被扫描多次。

所以，整个过程中表 t1 只会被全表扫描一次（即使是分块加入 join_buffer），而表 t2 被全表扫描的次数则是受限于 join_buffer 的大小。

注意，我们这里说全表扫描是基于磁盘的，如果数据已经加载到内存，其实就不算扫描了。

在 MySQL 官网中，也描述了一些 BNL 算法的特点，其中一些重要的点如下：

1. 只会将连接感兴趣的列存储在 join_buffer 中，而不是整行，这样 join_buffer 可以容纳更多的行记录。
2. join_buffer_size 系统变量确定用于处理查询的每个连接缓冲区的大小。
3. 为每个可以缓冲的连接分配一个缓冲区，因此可以使用多个连接缓冲区来处理给定的查询。
4. 连接缓冲区在执行连接之前分配，并在查询完成后释放。

最后，我们还要说一下 BNL 算法的一些不好的地方。

在使用 BNL 算法时，可能会对被驱动表做多次扫描，如果被驱动表是一个冷表，除了会导致 IO 压力大以外，更重要的是最后续查询的影响。

我们知道，在 Buffer Pool 中，由于 InnoDB 对 Bufffer Pool 的 LRU 算法做了优化，即第一次从磁盘读入内存的数据页，会先放在 old 区域，如果 1 秒后这个数据页不再被访问了，就不会被移动到 LRU 链表头部，这样对 Buffer Pool 的命中率影响就不大。

但是，如果一个使用 BNL 算法的 join 语句，多次扫描一个冷表，并且这个 SQL 执行时间超过 1 秒，就会将冷表的数据页移到 LRU 链表头部。

这种情况对应的是冷表的数据量小于整个 Buffer Pool 的 3/8，能够完全放入 old 区域的情况。

而如果这个冷表很大，就会出现另外一种情况：业务正常访问的数据页，没有机会进入 young 区域。

由于优化机制的存在，一个正常访问的数据页，要进入 young 区域，需要隔 1 秒后再次被访问到。但是，由于 join 语句在循环读磁盘和淘汰内存页，进入 old 区域的正常数据页，很可能在 1 秒之内就被淘汰了，没有机会进入 young 区域。这样，就会导致这个 MySQL 实例的 Buffer Pool 在这段时间内，young 区域的数据页都是冷表的数据，进而导致缓存命中率的降低，影响查询性能。

这两种情况都会影响 Buffer Pool 的正常运作。

所以，大表 join 操作虽然对 IO 有影响，但在 SQL 执行结束后，对 IO 的影响也就结束了。但是，对 Buffer Pool 的影响就是持续性的，需要依靠后续的查询请求慢慢恢复内存命中率。

为了减少这种影响，可以考虑增大 join_buffer_size 的值，减少对被驱动表的扫描次数。

也就是说，BNL 算法对系统的影响主要包括三个方面：

1. 可能会多次扫描被驱动表，占用磁盘 IO 资源；
2. 判断 join 条件需要执行 M * N 次对比（M、N 分别是两张表的行数），如果是大表就会占用非常多 CPU 资源；
3. 还可能会导致 Buffer Pool 的热数据被淘汰，影响缓存命中率。

## 优化 NLJ 和 BNL
在进行多表 join 时，如果可以使用 NLJ 算法，其实性能还是挺不错的，比在应用层拆分成多个语句然后再合并查询结果更方便。

但如果是使用 BNL 算法，那么在大表 join 时性能就差多了。

当然了，这两个算法都还有继续优化的空间，为了便于分析，还是创建两个表 t1、t2 进行分析。

```sql
CREATE TABLE t1 (
  id INT PRIMARY KEY,
  a INT,
  b INT,
INDEX ( a ));

CREATE TABLE t2 LIKE t1;

delimiter ;;
create procedure idata()
begin
  declare i int;
  set i = 1;
  while(i <= 1000)do
    insert into t1 values(i, 1001 - i, i);
    set i = i + 1;
  end while;
  
  set i = 1;
  while(i <= 1000000)do
    insert into t2 values(i, i, i);
    set i = i + 1;
  end while;
end;;
delimiter ;

call idata();
```

为了便于后面量化说明，在表 t1 里，插入了 1000 行数据，每一行的 a = 1001 - id 的值，即表 t1 中字段 a 是逆序的。同时，在表 t2 中插入了 100 万行数据。

### Batch Key Access
我们知道在二级索引范围查询时，如果查询的字段不在二级索引树，就需要多次回表，此时就可以基于 MRR 优化，将从二级索引树上查询到的主键集合在内存中做排序，统一向聚簇索引树进行回表，在大多数情况下，这样的操作可以将磁盘的随机读优化为顺序读。

而在 NLJ 算法中，也可以利用这种思想，所以就有了 Batch Key Access 的优化。

还是执行上面的 NLJ 算法的 SQL，如下：

```sql
select
  *
from t1 straight_join t2 on t1.a = t2.a;
```

NLJ 算法执行的逻辑是：从驱动表 t1，一行一行地取出 a 的值，再到被驱动表 t2 去做 join，也就是说，对于表 t2 来说，每次都是匹配一个值。

所以，我们可以从表 t1 里一次性地多拿一些行出来，一并传给表 t2。

前面我们知道 join_buffer 在 BNL 算法里的作用，是暂存驱动表的数据，在 BKA 算法中，也可以使用 join_buffer 来暂存驱动表的数据，一次性多传一些值给被驱动表，更好的利用 MRR 优化。

比如，在 join_buffer 中放入的数据是 1 ~ 100，表示的是在 t2 表 join 查询时需要的字段。当然，如果 join buffer 放不下 1 ~ 100 的所有数据，就会把这 100 行数据分成多段重复执行。

那么，这个 BKA 算法到底要怎么启用呢？

如果要使用 BKA 优化算法的话，需要在执行 SQL 语句之前，先设置

```bash
SET optimizer_switch='mrr=on,mrr_cost_based=off,batched_key_access=on';
```

其中，前两个参数的作用是要启用 MRR，因为 BKA 算法的优化要依赖于 MRR。

### BNL 算法的优化
关于 BNL 算法的多表 join，在执行 SQL 之前，就需要通过理论分析和查看 explain 结果的方式，确认是否使用了 BNL 算法。

如果确认优化器会使用 BNL 算法，就一定要做优化，常见做法是，给被驱动表的 join 字段加上索引，把 BNL 算法转成 NLJ 算法。

接下来，就具体看看，这个优化怎么做？

一些情况下，可以直接对被驱动表中的连接的字段建索引，这时就可以直接转成 NLJ 算法了。

但是，有时候确实会碰到一些不适合在被驱动表上建索引的情况，比如下面的 SQL：

```sql
select
  *
from t1 join t2 on t1.b = t2.b
where t2.b >= 1 and t2.b <= 2000;
```

表 t2 中存在 100 万行数据，但是经过 where 条件过滤后，需要参与 join 的只有 2000 行数据。如果这条语句同时是一个低频的 SQL，那么为这个语句在表 t2 的字段 b 上创建一个索引就很浪费了。

但是，如果使用 BNL 算法来 join 的话，SQL 的执行流程是这样的：

1. 把表 t1 的所有字段取出来，存入 join_buffer 中，这个表只有 1000 行，join_buffer_size 默认值是 256k，可以完全存入。
2. 扫描表 t2，取出每一行数据跟 join_buffer 中的数据进行对比，如果不满足 t1.b = t2.b，则跳过，如果满足 t1.b = t2.b，再判断其他条件，即是否满足 t2.b 处于 [1, 2000] 的条件，如果是，就作为结果集的一部分返回，否则跳过。

对于表 t2 的每一行，判断 join 是否满足时，都需要遍历 join_buffer 中的所有行。因此判断等值条件的次数是 1000 * 100 万 = 10 亿次。

在表 t2 的字段 b 上创建索引似乎没有必要，但是不创建索引这个 SQL 的执行又比较耗时，那么，有没有两全其美的办法呢？

这时候，就可以考虑使用临时表，大致思路是：

1. 把表 t2 中满足条件的数据放在临时表 tmp_t 中；
2. 为了让 join 使用 NLJ 算法，可以给临时表 tmp_t 的字段 b 加上索引；
3. 让表 t1 和 tmp_t 做 join 操作。

此时，对应的 SQL 语句的写法如下

```sql
CREATE TEMPORARY TABLE tmp_t(id int PRIMARY KEY, a int, b int, INDEX(b)) ENGINE=innodb;
INSERT INTO tmp_t SELECT * FROM t2 WHERE b >= 1 AND b <= 2000;
SELECT * FROM t1 JOIN tmp_t ON (t1.b = tmp_t.b);
```

可以看到，整个过程 3 个语句执行时间的总和相比于前面的 1 分 8 秒，性能得到了大幅提升。

1. 执行 insert 语句构造 tmp_t 表并插入数据的过程中，对表 t2 做了全表扫描，扫描行数是 100 万。
2. 之后的 join 语句，扫描表 t1，扫描行数是 1000，join 比较过程中，做了 1000 次带索引的查询。相比于优化前的 join 语句需要做 10 亿次条件判断来说，这个优化效果还是很明显的。

总体来看，不论是在原表上加索引，还是用有索引的临时表，我们的思路都是让 join 语句能够用上被驱动表上的索引，来触发 NLJ 算法，提升查询性能。

## 哈希连接
从 MySQL 8.0.18 开始，MySQL 对每个都含有等值连接条件的任何 join 查询都会采用哈希连接，并且其不依赖于索引，如果任何一对连接表不具备至少一个等值条件，则无法使用哈希连接，即采用较慢的 BNL 算法。

而从 MySQL 8.0.20 开始，连接不再需要包含至少一个等值连接条件才能使用哈希连接，所以 MySQL 不在支持 BNL 算法，全部被哈希连接锁替换，几乎所有情况，哈希连接都比 BLN 算法要快。

我们将 MySQL 升级到 8.0.20 之后的版本，再次执行上面的 SQL：

```sql
select
  *
from t1 join t2 on t1.b = t2.b
where t2.b >= 1 and t2.b <= 2000;
```

### 基本思想
哈希连接的基本思想是，将小表的数据加载到内存，并建立 hash 表，这样只需要对大表扫描一次，就可以完成 join 操作，输出匹配的记录。

如果小表能全部加载到内存自然更好，逻辑也简单，一般称这种 join 为 CHJ（Classic Hash Join），如果小表不能全部加载到内存，就需要分批进行加载，然后分批进行 join。

这是不是和 BNL 算法有点类似，都是先将小表的数据加载到内存，只不过哈希连接还会对小表的数据建立 hash 表，这样在遍历大表时，只需要在 O(1) 的复杂度内就可以与小表进行 join。

### In-Memory Join
哈希连接一般包括两个过程，创建 hash 表的 build 过程和探测 hash 表的 probe 过程。

#### build
在 build 时会选择较小表作为构建表（Build Table），然后据连接条件为构建表中的每一行计算一个哈希值，并将这些行存储在哈希表中。

### On-Disk Join
In-Memory Join 的限制条件在于，要求内存能容纳整个构建表。

在 MySQL 中，join 可使用的内存通过 join_buffer_size 参数控制，如果 join 需要的内存超出 join_buffer_size，那么 In-Memory Join 将无能为力。

为了解决该问题，MySQL 又引入了 **分区哈希连接**（Partitioned Hash Join）。这种方法通过将表分区，并使用磁盘临时存储分片数据，从而避免了需要多次扫描构建表的问题。

**分区阶段**（Partitioning Phase）：

+ **构建表分区**：使用哈希函数对构建表进行分区，每个分区数据会写到磁盘上的临时文件中，产生一次写 IO（将构建表的分片写入磁盘）。
+ **探测表分区**：使用相同的哈希函数对探测表进行分区，探测表也被写到磁盘上的临时文件中，也是一次写 IO（将探测表的分片写入磁盘）。

**分区连接阶段**（Partitioned Join Phase）：

+ 对每个分区编号的数据进行 In-Memory Join，由于分区后的数据量较小，通常可以在内存中完成。
+ **读取构建表分区**：从磁盘读取构建表的每个分区数据到内存中，这是第一次读 IO。
+ **读取探测表分区**：从磁盘读取探测表的每个分区数据到内存中，这是第二次读 IO。

通过分区哈希连接，MySQL 只需要一次扫描构建表和探测表，避免了在传统方法中需要多次扫描构建表的低效问题，将超出内存限制的大表分区处理，减少了内存占用。

## 小表驱动大表
了解了 MySQL 使用的 join 算法之后，我们应该就能理解大家常说的小表驱动大表了。

### NLJ
对于 NLJ 算法，我们假设大表需要扫描的行数为 M，小表需要扫描的行数是 N。

那么将大表作为驱动表，嵌套循环的执行复杂度可以近似看做 M * log N，而如果将小表作为驱动表，嵌套循环的执行复杂度可以近似看做 N * log M。

你可以基于 f(x) = log x / x 来做分析，它在 (e, +∞) 是一个单调递减函数，所以 M * log N 是大于 N * log M 的。

基于此，对于 NLJ 算法来说，将小表作为驱动表是合适的。

### BNL
对于 BNL 算法，就还需要考虑 join_buffer 的大小。

假设驱动表的行数是 N，被驱动表的行数是 M，在 join_buffer 足够大，无需分块的情况下：

1. 两个表都做一次全表扫描，所以总的扫描行数是 M + N；
2. 内存中的判断次数是 M * N。

这时候选择大表还是小表做驱动表，执行耗时是一样的。

在 join_buffer 不够，需要分块的情况下，假设驱动表需要分 K 块才能完成算法流程，那么

1. 需要对驱动表进行全表扫描，而被驱动表的扫描行数是 K * M，总的扫描行数是 N + K * M；
2. 内存判断 N * M 次。

而 K 又和 N 以及 join_buffer 的大小相关，假设 join_buffer 的大小为 λ，显然会有 λ * K = N。

所以总的扫描行数为 N + N / λ * M。

显然，内存判断次数是不受选择哪个表作为驱动表影响的，而考虑到扫描行数，显然选择小表作为驱动表，扫描的行数更少。

所以结论是，应该让小表当驱动表。

当然，如果能够增大 join_buffer_size 的大小，一次可以放入的行越多，分块也就越少，对被驱动表的全表扫描次数就越少。

这就是为什么，你可能会看到一些优化建议是，如果 join 语句很慢，尝试增大 join_buffer_size。

### 哈希连接
对于哈希连接而言，其实和 BNL 算法是一致的，因为本质都是将表数据读到内存中，所以我们也应该使用小表作为驱动表。

### 如何定义小表和大表
到这里我们就可以总结出来了，如果要使用 join，应该选择大表做驱动表还是选择小表做驱动表

1. 如果是 NLJ 算法，应该选择小表做驱动表；
2. 如果是 BNL 算法或者哈希连接：
    - 在 join_buffer_size 足够大的时候，是一样的；
    - 在 join_buffer_size 不够大的时候（这种情况更常见），应该选择小表做驱动表。

所以，这个问题的结论就是，总是应该使用小表做驱动表。

但是我们要如何定义小表和大表呢，难道仅仅是看表中的总记录数吗？

显然并不是这样。

如果给 SQL 加上 where 条件过滤，如下：

```sql
select
  *
from t1, t2 on t1.b = t2.b
where t2.id <= 50;
```

这样其实优化器应该选择的是 t2 作为驱动表，它是当前 SQL 条件下的“小表”，因为 join_buffer 只需要放入表 t2 的前 50 行，显然是更少的数据量。所以这里，“t2 的前 50 行”是那个相对小的表，也就是“小表”。

再来看另外一组例子：

```sql
select
  t1.b,
  t2.*
from t1, t2 on t1.b = t2.b
where t2.id <= 100;
```

这个例子里，表 t1 和 t2 都是只有 100 行参加 join。但是，这两条语句每次查询放入 join_buffer 中的数据是不一样的：

+ 表 t1 只查字段 b，因此如果把 t1 放到 join_buffer 中，则 join_buffer 中只需要放入 b 的值；
+ 表 t2 需要查所有的字段，因此如果把表 t2 放到 join_buffer 中的话，就需要放入三个字段 id、a 和 b。

所以，应该选择表 t1 作为驱动表。也就是说在这个例子里，“只需要一列参与 join 的表 t1”是那个相对小的表。

至此，我们应该明白，**在决定哪个表做驱动表的时候，应该是两个表按照各自的条件过滤，过滤完成之后，计算参与 join 的各个字段的总数据量，数据量小的那个表，就是“小表”，应该作为驱动表**。

## 要不要 join
最后，我们再来说说，这篇文章标题的问题，到底要不要 join？

在 MySQL 8.x 版本之前，如果查询执行计划时，发现 SQL 使用到了 BNL 算法，需要尽量将 BNL 算法优化为 NLJ 算法。

当然如果 SQL 本身就是要 NLJ 算法进行 join 那是没有问题的。

而在 MySQL 8.x 版本之后，由于 BNL 算法被更优秀的哈希连接替代，所以就算连接条件未使用索引其实性能也不算很差。

但如果有更好的方案，比如在内存中做关联，做一些数据冗余，或者基于一定的 join 关系，将数据库中多张表的数据打平做一张大宽表，可以考虑不使用 join。

总之，这是一个权衡的考量。
